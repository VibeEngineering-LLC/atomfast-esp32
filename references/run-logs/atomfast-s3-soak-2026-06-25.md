# Лог прогона — ESP32-S3-DevKitC-1 N16R8 (esp-idf) на AtomFast

**Прошивка:** `firmware/esp32-s3-devkitc/atomfast_gateway_s3.yaml` (esp-idf, scan 640/32, `web_server.log: true`)
**Плата:** ESP32-S3-DevKitC-1 **N16R8** (16 МБ Flash + 8 МБ embedded octal PSRAM, внешняя антенна через U.FL/IPEX, USB-C)
**Дата снятия:** 2026-06-25
**Метод:** read-only снимок `web_server` SSE-стрима `/events` живой платы. **Прошивка не выполнялась** — плата в продакшене, снят только текущий статус.
**Назначение:** приборное доказательство стабильности S3+esp-idf на AtomFast (закрывает прежнюю оговорку «перенос гипотезы с Radex-S3», README → «Итог: что выбрать»).

> Локальные идентификаторы (IP / MAC / SSID / web-auth) в этот публичный лог намеренно не включены.

## Непрерывность (фрагмент лога, ~45 с)

`uptime` инкрементируется монотонно (тот же boot, ребута нет), AGG-окна публикуются штатно:

```
{"title":"AtomFast S3 Gateway","ota":true,"log":true,"uptime":347880}
...
{"uptime":348065}
{"uptime":348071}
{"uptime":348081}
[I][atomfast:613]: AGG window=30s samples=14 cps=15.429 dose=0.0675 uSv/h
{"uptime":348091}
{"uptime":348101}
```

Во время съёмки: `0` строк `reboot` / `safe_mode` / `WARN` / `ERROR`; `web_server` отдаёт состояние без `json:111` (8 МБ octal PSRAM — запас heap снимает INC-05 даже при `log: true`).

## Вывод

ESP32-S3-DevKitC-1 N16R8 на **esp-idf** держит AtomFast-шлюз многосуточно без ребутов: BLE-сессии
многочасовые, реконнекты редкие и быстрые (нет `rsn 0x3e`-штормов INC-09 и `Auth Expired` INC-12,
характерных для esp-idf на классическом WROOM-32 без PSRAM). Это **приборное** подтверждение
рекомендации «S3 N16R8 + esp-idf», а не перенос вывода с Radex-S3.
